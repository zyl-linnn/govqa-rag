# GovQA-RAG：政务垂域大模型问答系统 · 详细设计文档

> 本项目是一个面向政务场景的检索增强生成（RAG）系统，基于 Qwen3‑8B 与 QLoRA 三阶段微调，支持口语化术语对齐、知识图谱增强检索、双重时效性校验与可追溯答案。  
> **当前版本**：设计文档先行，代码将于后续开放。

## 背景与目标

### 痛点
- 政务信息分散、术语晦涩（“办营业执照” vs “企业设立登记”）。
- 通用大模型存在幻觉、知识滞后、术语理解不稳定。
- 政策时效性强，过期文件误导风险高。

### 目标
- 构建一个**可迭代、可评估、可部署**的政务问答系统。
- 支持口语化输入、结构化知识召回、时效性校验与可追溯输出。
- 在通用、口语、推理三个维度上达到设计指标（≥89% / 75% / 80% 加权综合得分）。

## 系统架构

系统采用**五层模块化架构**，从上至下依次为：

```
┌─────────────────────────────────────────────────┐
│              用户交互层（CLI / Web）             │
├─────────────────────────────────────────────────┤
│  输入处理层：地域感知口语化术语对齐              │
├─────────────────────────────────────────────────┤
│  检索增强层：精确匹配 + 三元组检索 + 混合检索 + 重排序 │
├─────────────────────────────────────────────────┤
│  知识层：向量库 / 三元组问题库 / 元数据库 / 废止库  │
├─────────────────────────────────────────────────┤
│  模型层：Qwen3-8B + QLoRA (CPT→SFT→DPO)          │
├─────────────────────────────────────────────────┤
│  生成与后处理层：答案生成 + 时效性标注 + 参考文档列表 │
└─────────────────────────────────────────────────┘
```

### 数据流（离线 + 在线）
- **离线**：多源文档 → 清洗 → 向量化 → 存入知识库；三元组 → 问题生成 → 建立“问题‑问题”索引；手工构建元数据库与废止库。
- **在线**：用户问题 → 口语化改写 → 多路召回 → 重排序 → 注入时效性信息 → 模型生成 → 输出带引用的答案。

## 核心模块设计

### 3.1 多源数据处理与知识库

| 数据类型 | 来源示例 | 处理方式 | 规模 |
|---------|---------|---------|------|
| 结构化问答对 | Excel 政务问答 | 自动识别问题/答案列，清洗 HTML 标签 | 2,334 条 |
| 政策 PDF | 政府工作报告、最低工资通知 | pdfplumber 逐页提取 | 24 页 + 1 页 |
| 统计资料 PDF | 国民经济统计 | 表格解析生成指标描述 | 14 条指标 |
| Word 文档 | 清理通知、管理办法 | python-docx 提取文号 | 2 个文档 |
| 学校信息 | Excel | 每行转自然语言描述 | 635 条 |
| **总计** | | | **3,009 个文档对象** |

每个文档对象包含字段：`id`, `text`(摘要), `full_answer`(完整内容), `question`(如有), `source`, `doc_number`, `title`, `publish_date`, `file_path`。

### 3.2 地域感知口语化术语对齐

**设计目标**：将“我想给新车上牌，咋整？”转换为“我想办理机动车注册登记，如何操作？”

**两阶段流程**：
1. **候选检索（规则）**  
   - 加载地域术语库（山东：265条；获嘉：672条）  
   - 子串匹配，按口语词长度降序排序（避免短词覆盖长词）
2. **大模型决策与改写**  
   - 构造提示词（包含原始问题、候选替换项、地域信息）  
   - 调用 Qwen3-8B（temperature=0.3）生成改写结果  
   - 后处理：取第一句，去除多余前缀

**效果示例**：

| 地域 | 原始问题 | 改写后问题 |
|------|---------|------------|
| 山东 | 我想给新车上牌，咋整？ | 我想办理机动车注册登记，如何操作？ |
| 获嘉 | 办营业执照要啥材料？ | 企业设立登记需要哪些材料？ |

### 3.3 三元组增强检索

**动机**：结构化三元组（主语‑谓语‑宾语）与自然语言查询存在语义鸿沟。

**实现**：
- 从最低工资通知（22条）和统计资料（103条）中抽取三元组。
- 对每个三元组，调用 LLM 生成 3~4 个自然语言问题（单跳）。
- **多跳迭代**：对共享同一主语的多个三元组，生成复合问题（如“2026年2月12日发布的文件最低工资是多少？”）。
- 共得到 **191 个生成问题**，全部用 BGE 向量化，建立“问题‑问题”索引。
- 在线检索：用户问题向量与生成问题向量匹配（余弦相似度≥0.2），召回相关三元组。

### 3.4 混合检索与重排序

| 阶段 | 技术 | 输出数量 | 说明 |
|------|------|---------|------|
| 精确匹配 | 规则匹配文号/标题 | ≤ 文档总数 | 命中则分数=0.99 |
| 三元组检索 | 问题‑问题语义匹配 | ≤ 8 | 相似度阈值 0.2 |
| 混合检索 | BGE (0.6) + BM25 (0.4) | Top‑30 | 分数归一化后加权融合 |
| 重排序 | bge-reranker-v2-m3 | Top‑20 | Cross-Encoder 精细排序 |

### 3.5 三阶段模型微调（QLoRA）

基座模型：**Qwen3-8B**（4‑bit 量化 + LoRA）

| 阶段 | 数据集 | 规模 | 目的 | 训练损失变化 |
|------|--------|------|------|-------------|
| CPT（继续预训练） | BAAI/IndustryCorpus2 政务子集 | 20,000 条 | 适应公文风格与术语 | 2.27 → 2.08 |
| SFT（指令微调） | Government_Open_Data_Set（官方+口语） | 7,535 条 | 学会基于检索材料回答 | 2.53 → 1.56 |
| DPO（直接偏好优化） | 人工+AI 联合标注偏好对 | 180 对 | 提升安全性、完整性、时效性处理 | 0.70 → 0.64 |

**关键技巧**：
- DPO 中采用**轻度劣化**策略（保留语义但删除细节），避免过拟合。
- 独立加载 Policy 与 Reference 模型，使用验证集早停。
- 所有训练单卡可完成（CPT/SFT: V100 32GB；DPO: A800 80GB）。

### 3.6 双重时效性管理

- **第一重 – 废止文件库**  
  从清理通知中提取 6 条废止文号（含完整格式与简化格式），检索时命中则标记 `is_expired=True`。
- **第二重 – 文档元数据库**  
  手工维护 Excel（12 条核心记录），包含文号、标题、发布日期、废止状态等，通过 id 或前缀匹配注入文档对象。
- **答案输出**  
  在参考文档列表中明确标注“【已废止】”，并附文件路径供用户核对。

### 3.7 多轮对话管理

- 环形缓冲区保留**最近 8 轮**完整对话（用户问题、系统回答、引用文档 ID、时间戳）。
- 生成答案时，将最近 5 轮历史融入提示词，支持指代消解。
- 交互命令：`清空历史`、`查看历史`、`切换地域`。

## 实验设计与评估

### 测试数据集

| 子集 | 数量 | 特点 | 典型问题 |
|------|------|------|----------|
| 通用政务 | 20 | 常规问答，知识库覆盖充分 | “杭州市最低月工资标准是多少？” |
| 地域口语 | 20 | 含口语化表达，部分无直接对应 | “咋申请低保？” |
| 复杂推理 | 20 | 多跳、数值比较、条件约束 | “2026年2月12日发布的文件最低工资是多少？” |

### 评估指标体系

采用 **自动指标 + AI 评分** 混合方式，最终加权综合得分 =  
`准确性×0.40 + 完整性×0.20 + 流畅性×0.15 + 时效性×0.15 + 满意度×0.10`。

其中准确性 = `BLEU-4×0.05 + ROUGE-L×0.05 + BGE相似度×0.05 + (AI准确性评分/5)×0.85`。

### 消融实验组

| 实验组 | 模型 | 检索配置 | 目的 |
|--------|------|---------|------|
| Exp1 | Qwen3-8B（基座） | 无 RAG | 基线 |
| Exp2 | 基座 | 基础 RAG（BGE） | RAG 增益 |
| Exp3 | 基座 | Exp2 + 口语化对齐 | 术语对齐增益 |
| Exp4 | 基座 | Exp2 + 三元组增强 | 结构化知识增益 |
| Exp5 | 三阶段训练模型 | 完整系统（全部模块+重排序） | 最终性能 |

### 主要结果

| 指标 | Exp1 | Exp2 | Exp3 | Exp4 | Exp5 |
|------|------|------|------|------|------|
| 全测试集加权综合得分 | 28.5% | 71.8% | 70.2% | 71.3% | **83.8%** |
| 口语测试集加权得分 | 58.5% | 67.0% | **76.2%** | 68.5% | 81.5% |
| 推理测试集加权得分 | 28.0% | 79.8% | 75.3% | **82.8%** | 88.3% |

**结论**：
- RAG 是性能基石（+43.3%）。
- 口语化对齐模块在口语测试集上贡献 +9.2%。
- 三元组增强模块在推理测试集上贡献 +3.0%。
- 三阶段微调（尤其是 DPO）显著提升完整性、安全性和时效性处理。

## 技术栈与部署

| 类别 | 技术 |
|------|------|
| 基座模型 | Qwen3-8B |
| 微调框架 | HuggingFace transformers + PEFT (QLoRA) |
| 检索 | BAAI/bge-large-zh-v1.5 (稠密), rank_bm25 (稀疏), bge-reranker-v2-m3 (重排序) |
| 向量库 | Chromadb / FAISS（可互换） |
| 后端 | Python 3.10, Flask (API), Gradio (Web演示) |
| 训练硬件 | NVIDIA V100 32GB (CPT/SFT), A800 80GB (DPO) |
| 推理硬件 | V100 / T4 均可 |

> **推理性能**：当前在 V100 上单次问答约 10‑20 秒（模型生成占 6‑10 秒）。后续可引入 vLLM 加速。

## 快速开始（无代码阶段）

虽然代码尚未开放，但你可以按照以下步骤准备环境，后续直接运行：

1. **环境准备**  
   ```bash
   conda create -n govqa python=3.10
   conda activate govqa
   pip install torch transformers accelerate peft bitsandbytes
   pip install chromadb rank_bm25 sentence-transformers flask gradio
   ```

2. **下载模型**  
   从 HuggingFace 或 ModelScope 下载 Qwen3-8B 到本地。

3. **准备数据**  
   将你的政务文档按上述格式整理为 Excel/PDF/Word，并构建术语库。

4. **运行示例**（待代码发布）  
   ```bash
   python cli_demo.py --region 山东
   ```

> 当前阶段，建议先阅读本设计文档，了解整体架构。代码预计于 **2026年9月** 后开放。

## 未来工作与局限

### 局限
- 知识图谱构建半自动，三元组规模有限（125条）。
- 口语测试集中约 40% 问题无直接对应知识（依赖兜底策略）。
- 推理速度较慢（10‑20 秒/次）。
- 仅支持两个地区术语库，扩展需人工。

### 可改进方向
1. **自动化知识图谱构建**：从新政策文件中自动抽取三元组。
2. **术语库自动挖掘**：利用 LLM 从 12345 记录中提取口语‑标准术语对。
3. **推理加速**：vLLM、INT8 量化、算子融合。
4. **可视化界面**：Gradio / Streamlit 提供文件上传、对话历史可视化。
5. **细粒度时效性**：标识条款级修订而非整文废止。
